# Should fix

### 1. Centroid-only audio-only path still bypasses the new guard
- **Severity:** Medium (correctness gap in exactly the regime this commit claims to guard)
- **Where:** `framework/agents/term_memory/topic_router.py` `HybridWindowTopicRouter.observe` — `elif not has_text and has_probe and not audio_probe_guard.get("ok", False)`.
- **Why real:** The new guard is gated on `has_probe`. When audio-only (`not has_text`) has **no** positive probe (`has_probe` False) but a query embedding / EMA centroid is present, `_score_hybrid` renormalizes over the lone centroid signal so `conf ≈ normalized_top`, which can clear `min_confidence=0.60` — the exact confidence-inflation the guard exists to stop, just via the centroid instead of the probe. The repo's own `docs/auto_glossary_routing_probe_20260707.md` calls the centroid non-discriminative ("medicine windows closer to the NLP centroid"), so this is the signal most likely to false-switch. In the shipping config the probe usually returns positive cosine scores so `has_probe` is normally True; the gap bites only when the probe encode is empty/degenerate — narrow, but it is the same class of bug that was just "fixed."
- **How to verify:** Drive `observe` audio-only with `domain_probe_scores={}` and a strong wrong-domain `query_embedding` over ≥ consistency windows under production weights (`min_confidence=0.60`); check whether a switch fires with no raw-evidence gate.
- **Minimal fix direction:** Extend the guard trigger to the audio-only-no-text case regardless of `has_probe` (require an absolute evidence floor before any lone renormalized audio signal reaches switch confidence), or block audio-only switching entirely when `not has_probe`.

### 2. `min_positive_domains=2` couples the guard to "≥2 candidate domains scored positive"
- **Severity:** Low
- **Where:** `_audio_probe_guard` `ok` computation.
- **Why real:** The prior single-domain probe tests (`test_audio_only_probe_requires_three_consistent_windows`, `test_cached_probe_scores_can_confirm_audio_only_switch`) had to add a second `nlp` probe entry to keep passing, which shows the guard rejects a genuinely strong single-domain audio signal when only one candidate index is positive. Default config scores 4 domain slices so this is fine, but a deployment that preloads only the current + one target domain (`auto_glossary_preload_presets`) can silently make audio-only switching unreachable. Document the assumption or key the floor off the target/runner-up pair rather than a raw positive count.

### 3. Given measured ~0.01 real probe deltas, `min_raw_margin=0.08` likely makes audio-only switching inert in production
- **Severity:** Low (design/spec, documented)
- **Where:** guard threshold vs `docs/auto_glossary_routing_probe_20260707.md` ("Avg medicine top-score delta 0.0112 / -0.0461").
- **Why real:** The doc's own real-audio measurements put the medicine-vs-NLP raw top-score margin around/below 0.01–0.05, well under the 0.08 gate, so the guarded audio-only probe will almost never fire on real audio. Combined with the shipped scope having no ASR producer (text path also inert), the router effectively pins the initial preset in deployment. The docs acknowledge this ("remains a guarded fallback rather than the main production signal"), so this is not a bug, but the eval evidence (§ below) should not be read as demonstrating a working audio-only production switch.

# Performance risks
- Gating `query_window_embeddings` behind `return_query_window_embeddings` removes the prior unconditional per-request device→host copy for non-auto sessions — a small improvement, not a risk. `_audio_probe_guard` runs on every `observe` (including the text path) and does an O(#domains) sort/scan; with ≤4 domains this is negligible.

# Spec mismatch / surprise changes
- The documented fixed-weight formula (`0.60/0.25/0.10/0.05`) plus renormalization plus the new raw-evidence guard are now all reflected in `docs/adaptive_working_glossary.md`, and the switch-guard ordering doc lists the new guard. No undocumented behavior change in this diff.

Note the verification log commands still use bare `/mnt/data2/...` — harmless (log, not doc), but the cited doc path and the command that produced it differ by the `taurus/` prefix, so reproducibility depends on a symlink.

# Possible false positives (do not re-raise)
- The renormalized-confidence-clears-`min_confidence` concern from the prior pass is genuinely constrained for the audio-only *probe* path now: the guard requires raw `top_score ≥ 0.50`, raw margin `≥ 0.08`, ≥2 positive domains, and top-raw-domain == target, evaluated before the confidence check. `test_audio_only_contested_probe_does_not_false_switch_on_small_raw_margin` exercises the margin rejection.
- The `return_query_window_embeddings` gating is safe: the only consumer (`omni._process_batch` → `query_window_embeddings[idx]`) is itself gated on `auto_glossary_enabled`, which is what sets the flag; no other reader of `RetrievalResult.query_window_embeddings` exists in the packet, so non-auto sessions losing the field is not a regression.
- Guard fields default correctly in `RouterConfig`, so routers built in tests without passing them (e.g. `test_cached_probe_scores_can_confirm_audio_only_switch`) still enforce 0.50/0.08/2.
- `_audio_probe_guard` is computed before the if/elif chain but consumed only in the audio-only elif; the `unsupported_current` target reassignment does not corrupt it.

# Final recommendation
Mergeable with low risk as a scoped guard. Before merge, prefer to (1) close the centroid-only audio-only bypass (Should-fix #1) so no lone renormalized audio signal — probe *or* centroid — can clear `min_confidence` unguarded, and (2) add or explicitly defer an agent-level test that drives the `return_query_window_embeddings` + probe path through the real `MaxSimRetrievalPlugin`. The `min_raw_margin=0.08` vs measured ~0.01 deltas means the audio-only feature ships effectively inert; that is acceptable only because the docs frame it as a fallback and no eval claims otherwise — do not later cite the synthetic-probe fixture runs as evidence that audio-only production switching works.
