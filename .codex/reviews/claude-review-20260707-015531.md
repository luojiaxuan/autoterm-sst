# Must fix before merge

### 1. The required-meta schema is unverified and partly contradicts the documented event schema; the real run has never been exercised
- **Severity:** High (benchmark validity — harness likely non-functional or silently degraded on first real partial)
- **Where:** `eval/streaming_sst/eval_mixed_audio_switch.py` `validate_partial_meta_schema` (required top-level keys) and `extract_record`.
- **Why real:** `validate_partial_meta_schema` *requires* top-level `meta["cursor_samples"]`, `meta["router_text_source"]`, `meta["fixed_prompt_k"]`, `meta["candidate_pool_count"]` and raises `RuntimeError` on the first partial if any is missing. But the repo's own documented JSON event schema (`docs/adaptive_working_glossary.md`, reproduced in the packet's embedded review packets) shows `router_text_source` **nested under `topic_router.evidence.router_text_source`**, not at `meta` top level, and lists `prompt_reference_count`/`ui_reference_count` but **not** `cursor_samples`, `fixed_prompt_k`, or `candidate_pool_count` at all. If the server matches its own documented schema, the very first partial fails validation and the entire real run aborts before scoring anything. The only tests (`test_extract_record_...`) feed a hand-built event whose keys are constructed to match the harness's own assumptions, so they cannot catch a real schema mismatch. The dry-run never touches this path. The contract is therefore self-referential and unvalidated.
- **How to verify:** Grep the server/omni JSON-WS metadata emitter for `cursor_samples`, `router_text_source`, `fixed_prompt_k`, `candidate_pool_count`; capture one real partial from `127.0.0.1:8011` and diff its keys/nesting against `validate_partial_meta_schema`'s required tuple. (`omni.py`/server emitter are not in the packet, so this cannot be confirmed here — that is the point.)
- **Minimal fix direction:** Reconcile the required-key list and access paths against the actual emitted schema (e.g., read `router_text_source` from `topic_router.evidence` if that is where it lives), and validate against a captured real partial rather than a hand-built fixture.

# Should fix

### 2. `router_target_domain` reads only `to_domain`; the documented field is `to_preset` → wrong-switch detection can silently no-op
- **Severity:** Medium (silent pass, benchmark validity)
- **Where:** `extract_record` — `"router_target_domain": str(router.get("to_domain") or "")`; consumed by `summarize_run`'s `wrong_switches` filter (`... and item.get("router_target_domain") and router_target_domain != expected`).
- **Why real:** The documented `topic_router` payload shows `to_preset` (e.g. `"to_preset": "medicine_core_10k"`), not `to_domain`. If the server emits `to_preset`/a differently named domain field, `router_target_domain` is `""` for every record, the `and item.get("router_target_domain")` clause drops every candidate, `wrong_switch_count` is always 0, and the `not wrong_switches` term of `regression_pass` is vacuously satisfied — a false green. Unlike the top-level keys in #1, this field is not schema-validated, so it fails soft, which is worse.
- **Minimal fix direction:** Confirm the emitted target-domain field name and read it explicitly; if only a preset is emitted, map preset→domain before comparing.

### 3. Terminal-event type set is broad and may truncate the run early
- **Severity:** Low–Medium
- **Where:** `run_streaming_eval` — `if str(event.get("type")).lower() in {"final","done","complete","completed","eof"}: break`.
- **Why real:** If the server uses any of these words for a per-utterance / non-terminal event, the receive loop breaks early, truncates `records`, then blocks on `await feed_task` streaming the remaining (up to ~4.68 h of) audio to a socket nothing is reading. The server terminal-event name is unverified.
- **Minimal fix direction:** Match the exact terminal event the server emits, and cancel `feed_task` on early break instead of awaiting it.

### 4. `max_segs_per_talk` slices before existence filtering (seg-level regression of the fix in #1's talk-level version)
- **Where:** `read_acl_audio_blocks` — `selected = list(wavs)[:max_segs_per_talk]` then `[p for p in selected if Path(p).is_file()]`.
- **Why real:** Existence filtering happens *after* the per-talk slice, so if any of the first N segs are missing you silently get fewer than N segs (changing per-block audio length / span boundaries) even when later segs exist. Default `0` disables it, so low impact, but it is the same limit-before-existence class this branch just fixed at the block level.

### 5. `domain_probe_top_domain` silently yields `""` for a flat probe-score form
- **Where:** `domain_probe_top_domain` — non-dict `value` keeps `score=0.0`, so `rows[0][0] > 0.0` fails and it returns `""`.
- **Why real:** If the server ever emits `domain_probe_scores` as `{"medicine": 0.84}` (flat float), it passes `validate_partial_meta_schema` (only checks it is a dict) but produces empty probe-top for every event, making `probe_top_accuracy` silently `None`. Tie the extraction to the confirmed probe-score shape or validate the inner structure.

# Performance risks
- None on the serving hot path; the harness is client-side (no TP/PP, scheduling, or cache path). Operationally, `feed_sleep` defaults to `0`, so for the intended ~4.68 h run the feed task pushes the entire playlist as fast as the socket allows with `max_size=None` on the receive side. This relies entirely on TCP/WS backpressure; if the server eagerly drains the socket into an unbounded session audio buffer, this is a server-side memory risk, and it also makes the "switch latency in events" numbers reflect a non-streaming ingest pattern. Not a harness correctness bug, but note it before running long.
