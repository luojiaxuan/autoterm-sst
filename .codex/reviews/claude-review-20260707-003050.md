# Verdict
NEEDS_FIX

The router change (`has_text_topic_signal`) is small, targeted, and its stated fix is confirmed by the new unit test. But the benchmark that headlines this branch is driven by an oracle probe over gold reference text, so its central metrics are near-vacuous; the router change also silently alters a second, unguarded code path; and the new doc violates AGENTS path rules. The unit tests pass (17/17 per the log), but the reported 64-window/full-window JSON numbers are Taurus-only and not in the packet.

# Must fix before merge

### 1. Benchmark cannot detect the failure modes the task asks about (oracle probe + gold-reference proxy)
- **Severity:** Medium–High (benchmark validity / spec mismatch)
- **Where:** `eval/streaming_sst/eval_mixed_domain_switch.py` `probe_for_domain` (fed via `evaluate_playlist` with `probe_mode == "expected"`); `read_acl_blocks`/`read_medicine_blocks` window text.
- **Why real:**
  - `probe_for_domain(block.expected_domain)` always returns the *correct* domain at a fixed clean margin (0.35 vs 0.08). Every switch decision is therefore steered by an oracle that is never wrong. Consequently `wrong_switch_count == 0`, `steady_state_accuracy == 1.0`, and `regression_pass == true` are structurally guaranteed, not evidence that routing discriminates domains. The task explicitly asks to check "whether auto glossary routing can switch between NLP and medicine domains" and to not "hide wrong-domain steady-state behavior" — but with an always-correct probe there is no wrong-domain behavior to hide, so the metric is uninformative rather than validated.
  - The window text is ACL Chinese *target segments* and RASST medicine Chinese *references* — human gold translations, not model output produced under a possibly-wrong active glossary. This removes exactly the self-reinforcement risk repeatedly flagged in the branch's own prior reviews (`.codex/reviews/claude-review-20260706-234102.md` Must-fix #1). The proxy is therefore optimistic.
  - The doc `docs/auto_glossary_mixed_switch_20260707.md` `结论` states the router "可以...稳定切换...稳定" without inline caveat that this holds only under a per-block oracle probe; the mid-doc labeling ("probe_mode=expected 表示...可控期望域证据") is present but the headline conclusion and the `adaptive_working_glossary_eval.md` addition overstate it.
- **How to verify:** Add a `probe_mode` variant that feeds contested/noisy multi-domain probe scores (wrong domain occasionally higher, margins near the `generated_target_probe_min_raw_margin=0.01` floor) and confirm whether `wrong_switch_count` can become nonzero; grep `probe_for_domain` to confirm it is keyed off `block.expected_domain` (it is).
- **Minimal fix direction:** Either add a noisy/contested-probe scenario and report it, or restate the doc conclusion as "state-machine switches correctly *given* oracle domain-probe evidence," making explicit that domain discrimination and self-reinforcement are untested by this proxy.

### 2. `_score_hybrid` now renormalizes on `has_text_topic_signal` while `observe` still gates on `has_text` — reopens the probe-saturates-confidence path for text-present/topic-empty windows without a raw-evidence floor
- **Severity:** Medium (hidden cross-path robustness regression)
- **Where:** `framework/agents/term_memory/topic_router.py` — new `has_text_topic_signal` used in `active_weight_sum`, the weighted sum, and `alpha` (diff hunks @764/@780/@796/@807); the `observe` guard chain still computes its own `has_text` and only triggers `_audio_probe_guard` on `not has_text and has_probe`.
- **Why real:** For a `manifest_source`/`streaming_asr` window whose text has zero topic-keyword hits (`has_text_topic_signal=False`) but a probe present, the confidence is now renormalized over probe-only weight (denominator collapses to the probe weight), driving `conf` toward the normalized probe top — the exact saturation `_audio_probe_guard` (floors 0.50/0.08/2) was added to prevent. But because `observe` still sees `has_text=True`, the audio-probe guard branch is skipped and only the 2-window `min_consistent_windows_with_text` applies. `generated_target` is protected by `_generated_target_probe_guard`, but its floors are lenient (top 0.25, margin 0.01), so a weak/contested probe on generic generated text can also clear it. The new unit test uses the clean oracle probe (0.35/0.08) and cannot surface this.
- **How to verify:** Drive `observe(..., router_text=<generic text, no keyword hits>, router_text_source="manifest_source", domain_probe_scores=<contested probe, medicine 0.30 vs nlp 0.28>)` over ≥2 windows under production weights (`min_confidence=0.60`) and check for a switch with no raw-evidence floor; repeat with `generated_target` and a probe just above the 0.25/0.01 floor.
- **Minimal fix direction:** Reconcile the two definitions — either apply an absolute-evidence floor whenever the *effective* signal is probe-only (regardless of `has_text`), or route the `has_text && !has_text_topic_signal` case through the same guard as audio-only. Add a contested-probe regression asserting no false switch.

# Final recommendation
Do not merge as-is. Before merge: (1) stop presenting the oracle-probe/gold-reference run as demonstrating domain switching — add a contested/noisy-probe scenario or restate the conclusion as "state machine given correct probe evidence"; (2) reconcile `has_text` vs `has_text_topic_signal` so a probe-only-effective window cannot clear `min_confidence` without a raw-evidence floor on the `manifest_source`/`streaming_asr` path, and add the corresponding negative test; (3) host-qualify the new doc's paths per AGENTS. The core router edit is correct for keyword-bearing text; the risk is entirely in the topically-empty sub-case and in the benchmark's evidentiary weight.
