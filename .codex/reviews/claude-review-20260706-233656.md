The change under review is `368472d → 7449b53` (`Add E2E generated-target glossary routing` + `Harden generated-target glossary routing`). It fills in the previously no-op `_after_translation_tick` to write generated target text into `router_text_window`/`router_text_source`, drops `common_terms` from the default active inventory (`base_slice: none`, `type: diagnostic`), simplifies `_autoterm_base_preset`, adds a stricter `min_consistent_windows_generated_target=3` path, and adds Chinese topic keywords for **all four** domains. # Must fix before merge

### 1. The "one routing tick behind" / no-self-influence property is unverifiable from the packet and is the correctness hinge
- **Severity:** High if the call ordering is wrong.
- **Where:** `framework/agents/omni.py` `_after_translation_tick` (writes `router_text_window`) vs `_observe_active_glossary` (reads it). The call site of `_after_translation_tick` inside the streaming/batch loop is **not in the packet** (`omni.py` truncated).
- **Why real:** `docs/adaptive_working_glossary.md` asserts "The generated target window is one routing tick behind the current chunk, so a chunk's own translation cannot influence its own glossary retrieval." That holds only if `_after_translation_tick` runs strictly *after* the per-chunk `_observe_active_glossary`. If it runs before (or in the same routing pass), the current chunk's own translation — produced under the possibly-wrong active glossary — feeds that chunk's domain decision, closing a self-reinforcing loop with no guardrail. The added test `test_generated_target_text_is_used_by_next_router_tick_only` hand-sequences `observe → after_tick → observe`; it proves the flow-through in mock but does **not** prove the production loop orders the two calls that way.
- **How to verify:** `grep -n "_after_translation_tick" framework/agents/omni.py`; confirm it is invoked strictly after the per-chunk `_observe_active_glossary`. Add a test that drives the real per-chunk path and asserts a chunk's translation only affects the *next* chunk's `router_text_window`.
- **Minimal fix direction:** None if ordering is already correct; otherwise move the write after routing.

### 2. `_after_translation_tick` assumes `session.history` holds target translations; this is unconfirmed
- **Severity:** High if `history` holds source/messages.
- **Where:** `_after_translation_tick` — `texts = [... for item in session.history[-window:] ...]`.
- **Why real:** The whole feature depends on `session.history` containing *emitted target translations* (Chinese). If it holds source-side text, message dicts, or interleaved source+target, `text_topic_score` is computed over the wrong content and the generated-target route is silently miscalibrated. `OmniSession.history: List[str]` is declared but what is appended to it is not in the packet.
- **How to verify:** Inspect every `session.history.append(...)` in `omni.py`; confirm it appends the target translation string at the point `_after_translation_tick` reads it, and that the caller's append vs the internal dedup (`texts[-1] != current_text`) don't double-count.

# Should fix

### 3. Several new Chinese keywords are high-frequency/dual-use → precision loss biased toward the default NLP slice
- **Where:** `domain_taxonomy.py` new entries, e.g. nlp `提示|微调|解析` (`提示`="hint/indicate" is common in general and clinical Chinese: "结果提示…"; `解析` is generic), medicine `试验` (generic "trial/experiment"), medicine `核磁|磁共振|CT` marked `case_sensitive=True` (irrelevant for CJK, harmless but sloppy).
- **Why real:** Because the default active slice is `nlp_core_10k`, spurious NLP keyword hits combine with `min_current_margin` (which favors staying) to make it both easier to falsely retain NLP on non-NLP audio and harder to leave. No test exercises noisy/realistic Chinese generated text — only hand-built keyword-dense clean sentences — so precision loss cannot surface.
- **Minimal fix direction:** Drop the most generic tokens or lower their weights; add a noisy-Chinese ACL-only regression asserting `switch_count == 0`.

### 4. Renormalized single-signal confidence still saturates `min_confidence` on the generated-target-only path
- **Where:** `topic_router.py` `_score_hybrid` renormalization (introduced earlier on this branch) + `observe`: `generated_target ⇒ has_text=True`, bypassing `_audio_probe_guard`.
- **Why real:** In the shipping config the generated-target text is often the only active text signal; renormalization drives `conf` toward its top-normalized value, so ~3 windows of generic keyword hits can clear `min_confidence=0.60`. The 3-window consistency guard is the *only* added protection for a signal the task explicitly flags as self-biased; there is no raw-evidence floor as there is for the probe path. Given items 3 and the default NLP bias, this is where a false switch would appear.
