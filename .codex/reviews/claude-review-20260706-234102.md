# Must fix before merge

### 1. Generated-target routing has no raw-evidence floor; the self-reinforcement guard the task asked for is not actually implemented
- **Severity:** Medium–High (robustness / spec of the feature's whole novelty)
- **Where:** `framework/agents/term_memory/topic_router.py` `HybridWindowTopicRouter.observe` — `is_generated_target` only selects `min_consistent_windows_generated_target` (3); the `_audio_probe_guard` (`not has_text and has_probe`) is bypassed for generated target because `has_text=True`.
- **Why real:** The task explicitly asks "whether using generated target text can self-reinforce wrong glossary choices without enough guardrails." The only added protection over trusted source text is one extra consistency window (3 vs 2). Generated target text is produced under the currently-active (possibly wrong) glossary, so its keyword content is biased toward the current domain. Unlike the audio-only probe path — which requires a raw top-score ≥ 0.50, raw margin ≥ 0.08, and ≥2 positive domains before a renormalized score may switch — the generated-target path has no analogous absolute-evidence floor. Because the branch upstream renormalizes over present signals (documented in `docs/adaptive_working_glossary.md`), a single active text signal can drive `conf` toward its top-normalized value and clear `min_confidence=0.60`, so ~3 windows of biased keyword hits can switch, and `min_current_margin` simultaneously makes it *harder to leave* a wrong initial slice. The dominant realistic failure is sticky retention of the wrong (default `nlp`) slice, which no guard or test addresses.
- **How to verify:** Drive `observe(..., router_text_source="generated_target")` from an `nlp` start with target text that echoes NLP terms while the underlying content is medicine; confirm whether the router can (a) falsely retain nlp or (b) switch on generic biased hits. No such regression exists.
- **Minimal fix direction:** Give `generated_target` an explicit raw-evidence/margin floor before it can reach switch confidence (parallel to `_audio_probe_guard`), and/or add a realistic noisy regression asserting the intended behavior; do not rely solely on the 3-window count.

### 2. The one-tick-behind / no-self-influence property is guaranteed only by a brittle source-substring test
- **Severity:** Medium (correctness hinge, weakly protected)
- **Where:** `test_auto_working_fixed_top10.py::test_streaming_loop_orders_routing_before_generated_target_write`; the actual streaming-loop call sites in `framework/agents/omni.py` are truncated in the packet.
- **Why real:** The docs assert "a chunk's own translation cannot influence its own glossary retrieval," which holds only if `_observe_active_glossary` (inside `_retrieve_batch`) runs before `_after_translation_tick` (inside `_generate_one`/`_generate_batch`) for the same chunk. The added test enforces this via `source.index(...)` string-position comparisons — a structural heuristic that breaks silently on any refactor that reorders code textually without changing runtime order, and cannot observe the actual control flow. It also does not prove `_after_translation_tick` is invoked *only* in the generate path, nor that `session.history` contains only emitted target translations (it is only inferred from `append(text)` preceding `after_translation_tick(text=text)`).
- **How to verify:** Confirm from the untruncated loop that observe precedes after-tick per chunk, and add a runtime test (mock loop) asserting chunk N's translation only affects chunk N+1's `router_text_window` — the hand-sequenced `test_generated_target_text_is_used_by_next_router_tick_only` calls the methods manually and does not exercise the loop.
- **Minimal fix direction:** Add a mock end-to-end streaming assertion; keep the string test only as a supplement.
