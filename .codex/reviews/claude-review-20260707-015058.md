# Must fix before merge

### 1. `cursor_samples` fallback silently fabricates audio-position alignment
- **Severity:** High (benchmark validity)
- **Where:** `eval/streaming_sst/eval_mixed_audio_switch.py` `run_streaming_eval`, `cursor_samples = int(meta.get("cursor_samples") or (events_seen * chunk_samples))`.
- **Why real:** Every downstream metric — `expected_domain_at`, `domain_transitions` (boundary crossing, `latency_events`, `latency_s`), steady-state accuracy, probe accuracy — is keyed off `cursor_samples`. If the server does not emit `meta.cursor_samples`, the code substitutes `events_seen * chunk_samples`. `events_seen` counts *partial events*, not audio chunks fed; the server emits partials on its own generation cadence (typically one per `latency_multiplier` segments, not one per sent chunk), so this product has no relationship to the true audio cursor. The whole playlist-span alignment then drifts monotonically and the reported switch latencies / steady-state accuracy are noise. Critically, the documented JSON event schema in `docs/adaptive_working_glossary.md` (in this packet) lists `topic`, `topic_router`, `domain_probe_scores`, `retrieve_s` — but **not** `cursor_samples`, so the fallback is the *likely* path, not the edge case. A legitimate `cursor_samples == 0` on the first event also wrongly triggers the fallback (`0 or ...`).
- **How to verify:** Grep the server/omni JSON-WS metadata emission for `cursor_samples`; run one short real chunk against `127.0.0.1:8011` and inspect whether `meta.cursor_samples` is present and monotonically increases with fed samples.
- **Minimal fix direction:** Require `cursor_samples` explicitly (`meta.get("cursor_samples")` is `None` → raise / record a hard failure), or track cursor on the *feed* side (bytes actually sent) instead of deriving it from event count. Do not silently substitute an event-count proxy.

# Should fix

### 2. Metric extraction assumes an unverified server-metadata contract and degrades silently
- **Where:** `run_streaming_eval` record construction: `router.get("to_domain") or router.get("target_domain")`, `meta.get("router_text_source")`, `meta.get("candidate_pool_count")`, `meta.get("fixed_prompt_k")`, `topic.get("active_domain")`, etc.
- **Why real:** Only `topic.active_domain`, `topic.active_glossary_preset`, `topic.switch_count`, `topic_router.action`, `domain_probe_scores`, `retrieve_s`, `domain_probe_s` are corroborated by the documented schema. If `to_domain`/`target_domain`, `router_text_source`, `candidate_pool_count`, or `cursor_samples` are named differently server-side, each defaults to `""`/`0` and `wrong_switch_count` is undercounted (the `and item.get("router_target_domain")` filter drops empties), `router_text_sources`/candidate-count diagnostics become empty — with `regression_pass` still able to read `true`. None of this fails loudly. The tests never exercise this mapping (see Missing tests).
- **Minimal fix direction:** Assert the required meta keys exist on the first real partial and error out on the schema mismatch, rather than defaulting.

### 3. Sole run-termination is a single 60 s idle timeout — fragile for the intended 4.68 h run
- **Where:** `run_streaming_eval` `idle = 0; while idle < 1: ... except asyncio.TimeoutError: idle += 1`.
- **Why real:** The loop exits on the *first* 60 s gap with no message. For a ~4.68 h real playlist with model-generation overhead and variable per-segment latency, a >60 s stall mid-stream would exit early, truncate `records`, and mark later transitions as failed. There is no completion/EOF-final-event detection (only `type == "partial"` is counted). After early exit, `await feed_task` keeps sending audio while nothing reads the socket, and `max_size=None` gives the client an unbounded receive buffer.
- **Minimal fix direction:** Terminate on an explicit server end/complete event and/or a bounded idle count after EOF; cancel/await `feed_task` deterministically.

### 4. Audio wire-format is unverified
- **Where:** `feed()` sends `chunk.tobytes()` where `chunk` is `float32` in [-1,1) (`read_wav` divides int16 by 32768). If the JSON-WS input path expects int16 PCM, every byte is misinterpreted. Not confirmable from the packet.
- **Minimal fix direction:** Confirm the server's expected PCM dtype against the existing `eval_auto_glossary.py` audio path and match it explicitly.

### 5. `read_acl_audio_blocks` applies `limit_items` before file-existence filtering
- **Where:** `list(by_talk.items())[: limit_items]` then `selected = [p for p in selected if Path(p).is_file()]`; a block with zero existing files is dropped.
- **Why real:** If any of the first `limit_items` talks has missing wavs, you silently get fewer than the intended 5 ACL blocks (or 0), changing playlist length/transition count without warning. The dry-run happening to find 5 blocks does not guarantee this on other checkouts.
- **Minimal fix direction:** Filter for existence first, then take `limit_items`.

# Final recommendation
Do not merge as-is. Before merge: (1) make `cursor_samples` a hard requirement or derive the cursor from bytes fed, so span alignment cannot be silently fabricated; (2) assert the required `meta`/`topic`/`topic_router` field names on the first real partial instead of defaulting to empty; (3) replace the single-60 s-idle termination with explicit completion detection for the long real run; and (4) confirm the PCM wire dtype and fix the ACL block-limit-before-existence ordering.
