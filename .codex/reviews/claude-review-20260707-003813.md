# Verdict
NEEDS_FIX

The router edit (`has_text_topic_signal`) does close the specific dilution weakness it targets and the new unit tests confirm that narrow case. But the same edit silently changes the renormalization denominator on a second, unguarded code path, reintroducing a probe-free false-switch route; and the headline benchmark's "pass" numbers are structurally produced by a per-block oracle probe, so they do not demonstrate the domain discrimination the task asks about. The deployable `generated_target` path is safe; the regression is on the `manifest_source`/`streaming_asr` routes and is not covered by the benchmark (which never supplies query embeddings) or any test.

Evidence bounds: the four `_score_hybrid` hunks (@764/@785/@801/@812) and the `observe` guard hunk (@596) are shown, so Finding A rests on shown code. `_normalize_score_map`, the full `_probe_guard`, and the JSON artifacts are truncated/absent; the 640/1905-window numbers are Taurus-only and unverifiable from the packet.

# Must fix before merge

### 1. `has_text_topic_signal` collapses the renormalization denominator on the text-present/topic-empty path, letting a lone speech centroid saturate `min_confidence`
- **Severity:** High (hidden cross-path regression; directly touches the "text alone must not switch without probe" acceptance criterion, on a non-`generated_target` source)
- **Where:** `framework/agents/term_memory/topic_router.py` `_score_hybrid` — `active_weight_sum` (@785), the weighted sum (@801/@812), and `alpha` (@812) now key off `has_text_topic_signal` instead of `has_text`; `observe`'s new guard (@596) only fires on `has_probe and has_text and not has_text_topic_signal`.
- **Why real:** For a `manifest_source`/`streaming_asr` window whose text has zero topic-keyword hits (`has_text_topic_signal=False`), no probe (`has_probe=False`), but a query embedding / EMA centroid present:
  - **Before:** `has_text=True` kept `text_topic_weight` (0.60) in `active_weight_sum`; with `text_score=0` for all domains, `conf = 0.10·centroid / (0.60+0.10) ≈ 0.14·centroid < 0.60` → blocked by `min_confidence`.
  - **After:** text weight is removed from the denominator, so `conf = 0.10·centroid / 0.10 = centroid_normalized`, which saturates toward ~1.0 and clears `min_confidence=0.60`. The new `probe_only_evidence_insufficient` guard is gated on `has_probe`, so this centroid-only, no-probe sub-case falls straight through to the confidence check and can switch after `min_consistent_windows_with_text=2`. The docs' own probe note calls the centroid non-discriminative, so this is a likely false switch, and it happens with **no probe evidence at all** — the opposite of what the guard was added to enforce.
- **How to verify:** `observe(state, query_embedding=<wrong-domain vector>, [], now_s=k, router_text="<generic text, no keyword hits>", router_text_source="manifest_source", domain_probe_scores={})` over 2 windows under the shipped weights (`min_confidence=0.60`); confirm a switch fires post-change and did not pre-change. Note the mixed-switch benchmark cannot surface this: `evaluate_playlist` always calls `router.observe(..., None, ...)` (query_embedding=None → `has_embedding=False`), so `active_weight_sum` is 0 in the no-probe row and no switch occurs.
- **Minimal fix direction:** Apply the raw-evidence floor whenever the *effective* signal degrades to a single renormalized signal — route `has_text and not has_text_topic_signal` (regardless of `has_probe`) through the same absolute-evidence guard as the audio-only path, or block a lone renormalized centroid from reaching switch confidence. Add the corresponding negative test.

# Should fix

### 2. Benchmark "pass" is steered by a per-block oracle probe; expected-mode results don't demonstrate domain discrimination
- **Severity:** Medium (benchmark validity)
- **Where:** `eval/streaming_sst/eval_mixed_domain_switch.py` `probe_for_domain(block.expected_domain, mode="expected")`; window text is ACL gold target segments + medicine gold references.
- **Why real:** In `expected` mode the probe is always the correct domain at a fixed clean margin (0.90 vs 0.35), fed from the very first window of each new block. `wrong_switch_count=0`, `steady_state_accuracy=1.0`, `regression_pass=true` are therefore near-structural, and the `inverted` row (probe wrong ⇒ 10 wrong switches) confirms the *probe* dominates the decision — so the correct outcomes in `expected` mode are probe-driven, not evidence that target text (or real MaxSim audio) discriminates domains. The gold references also remove the self-reinforcement risk that a model-generated target would carry. The doc's second paragraph now caveats this ("不证明真实 speech domain probe 的域判别质量"), which is a genuine mitigation over the prior pass, but the `结论` and the `adaptive_working_glossary_eval.md` addition still read as a capability claim ("稳定切换").
- **Minimal fix direction:** State the conclusion as "state machine switches correctly *given* clean probe evidence"; do not present the oracle-probe/gold-reference run as evidence of domain discrimination or self-reinforcement resistance.

# Performance risks
None material. The change is CPU-side scalar/weight bookkeeping in `_score_hybrid`; the benchmark runs the router in-process with hysteresis disabled and no retriever/GPU, TP/PP, scheduling, or cache path.

# Spec mismatch / surprise changes
- `_score_hybrid` weight/`alpha` semantics silently change for the topically-empty text case (`manifest_source`/`streaming_asr`): text weight dropped from the denominator, `audio_ema_alpha` used instead of `text_ema_alpha`. The docs describe the fixed-weight formula and the audio-probe floor but not this sub-case (Finding A).
- Benchmark presents an oracle-probe/gold-reference proxy; the `no-probe` row only re-confirms `generated_target_probe_guard` blocks without probe evidence — it does not show target text alone identifies the domain (Finding 2).

# Possible false positives (do not re-raise)
- The `generated_target` deployable path is **not** affected by Finding A: it is blocked at the prior branch (`generated_target_probe_evidence_insufficient`) whenever probe evidence is absent (`min_positive_domains=1`), so a probe-free generated-target window cannot reach the renormalized confidence check.
- For keyword-bearing text (`has_text_topic_signal=True`), the change is identical to the old `has_text` behavior; the existing `test_generated_target_chinese_*` / `test_acl_to_medicine_*` / `test_metadata_prior_*` cases are unaffected (20/20 pass per the log).
- The transition grace window equals `max_switch_windows` (3), which exactly covers the ≤3-window consistency lag; a switch slower than 3 windows fails the transition and its excess wrong windows count in steady-state — so the grace design does not hide steady-state error (acceptance criterion met on this point).
- `.codex/` is untracked (`?? .codex/`) and not part of the 4 reviewed commits; the diff stat touches only the 6 tracked files, so it is not an AGENTS "commit generated dumps" violation.
- `build_schedule` uses a locally seeded `random.Random(seed)`; random-schedule reproducibility is fine.

# Final recommendation
Do not merge as-is. (1) Reconcile `has_text` vs `has_text_topic_signal` so a window whose *effective* evidence is a single renormalized signal — centroid or probe — cannot clear `min_confidence` without an absolute-evidence floor, and add the negative test for the `manifest_source`/`streaming_asr` + centroid + no-probe case. (2) Restate the benchmark conclusion as "state machine given clean probe evidence," not domain discrimination, since every `expected`-mode pass is oracle-steered. (3) Document/align the new default data paths and prefer `/mnt/taurus/data2` per AGENTS. The narrow dilution fix and its unit tests are correct; the risk is the untested topic-empty sub-case and the benchmark's evidentiary weight.
