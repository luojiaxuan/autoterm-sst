### 1. `_prompt_references` guards only `"none"`, but the criterion names two empty sentinels ("no_glossary/none")
- **Severity:** High if a second sentinel exists; otherwise evidence gap.
- **Where:** `framework/agents/omni.py:1737` `_prompt_references` — `should_force_fixed_k = ... or ((session.glossary_preset or "").strip() and (session.glossary_preset or "").strip() != "none")`.
- **Why real:** The task says "no_glossary/none remains empty," yet the code only excludes the literal `"none"`. Empty string / `None` are handled by the truthiness check, but a literal `"no_glossary"` (or any non-`"none"` sentinel the UI/config uses for "no glossary") would be treated as a real preset and get `force_exactly_k_references` backfill — injecting synthetic filler terms into a prompt that must stay empty. I cannot confirm from the packet whether `DEFAULT_GLOSSARY_PRESET`, the UI, or `glossary.py` ever set anything other than `"none"` for the empty case.
- **How to verify:** `grep -rn "no_glossary\|DEFAULT_GLOSSARY_PRESET\s*=\|glossary_preset\s*=" framework/agents/glossary.py framework/agents/omni.py serve/` and enumerate every value assigned to `session.glossary_preset` for the "no glossary" selection.
- **Minimal fix direction:** Guard against the actual set of empty sentinels (e.g. a shared constant/allowlist), not a single hardcoded `"none"`. Add a test that the alternate sentinel stays empty.

### 2. The guard change alters the live serving prompt, and nothing measures the effect
- **Severity:** Medium–High (scope/spec + unmeasured regression).
- **Where:** `framework/agents/omni.py:1740` — the branch now fires for any non-none fixed preset, so a fixed-preset session with fewer than `k` retrieved terms is now backfilled to exactly 10 with `fixed_prompt_k_domain_neutral_default` filler terms.
- **Why real:** The criterion was framed for benchmark parity, but this code path is the production/demo serving path, not eval-only. Per the doc itself, old fixed runs averaged 1.05–2.66 prompt refs; after this change every fixed-preset user's prompt is padded with 7–9 synthetic neutral terms. That changes what the model sees on live translations. The only coverage is length-invariant (`len == 10`); there is no term_acc / translation-quality run on the *new* serving path. The diagnostic table in the doc (5/6, 3/6, 5/6) is recomputed from **pre-change** JSON outputs, so it validates the old behavior, not the code being merged. A serving-behavior change ships with zero metric-level evidence.
- **How to verify:** Re-run `score_mixed_audio_terms.py` on outputs generated with the patched serving code (the doc says this is required future work); compare fixed-preset term_acc before/after backfill.
- **Minimal fix direction:** Either gate the forced-k backfill to the eval/benchmark path via an explicit typed flag, or accept the live change but attach a term_acc run on post-change outputs before claiming parity. Do not present the pre-change diagnostic as validating the new path.
