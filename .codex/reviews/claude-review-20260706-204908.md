# Verdict
NEEDS_FIX

The change under review is `6d736b5 → 8a911aa` ("Preserve window probe routing semantics"): it (a) carries per-window speech embeddings from the main retrieval pass into the domain probe so probe scoring is genuine multi-window MaxSim instead of a mean-pooled dot product, (b) renormalizes the router score over the signals actually present in a window, and (c) makes `common_10k` an explicit working preset. (a) and (c) look correct and are tested. (b) is a real semantic change to the documented score/confidence gate that amplifies a signal the repo itself calls "too noisy," and the regression that would expose a resulting false switch is not exercised. There is also an AGENTS-violating/self-inconsistent doc path.

# Must fix before merge

### 1. Weight renormalization inflates single-signal confidence toward 1.0, weakening the `min_confidence` gate on the audio-only probe path
- **Severity:** Medium–High (correctness/robustness of the switch decision; hidden regression risk)
- **Where:** `framework/agents/term_memory/topic_router.py` `_score_hybrid` — `active_weight_sum` and `conf = weighted / active_weight_sum`.
- **Why real:** When only one signal is active, `active_weight_sum` equals that signal's own weight, so `conf = weight*normalized_top / weight = normalized_top`. In the shipping (no-ASR) config `router_text_source == "none"`, so the audio-only regime routinely has just the probe active (or probe + a weak centroid). A lone, top-normalized probe therefore drives `conf` to ~1.0 and clears `min_confidence=0.60` — the exact gate the README/docs describe as "minimum router confidence to switch." This is precisely what made audio-only switching reachable (the prior pass's `0.40 < 0.60` complaint), but it does so by amplifying the domain-probe signal that `docs/auto_glossary_routing_probe_20260707.md` characterizes as "too noisy by itself" (medicine-vs-NLP top-score delta ~0.01). The only remaining guards are `min_margin=0.15` and the 3-window audio-only consistency streak; the confidence gate no longer meaningfully constrains audio-only switches.
- **How to verify:** Drive `_score_hybrid`/`observe` audio-only (no `router_text`) with a *multi-domain* probe dict where the wrong domain is marginally higher on a few consecutive windows (the ACL-window-predicts-medicine 3/13 case the probe doc reports), using production weights and `min_confidence=0.60`. Confirm whether a false switch fires. The existing `test_audio_only_probe_requires_three_consistent_windows` feeds a single-domain probe dict (`{"medicine": ...}` only), which yields an artificial full margin and cannot surface this.
- **Minimal fix direction:** Either keep the renormalization but reinstate an absolute-evidence floor on the probe-only regime (e.g., require raw probe `top_score` / margin above an explicit threshold before it can reach switch confidence), or add an audio-only ACL-only regression with a realistic multi-domain noisy probe asserting `switch_count == 0`. Do not present a renormalized single-signal score as "confidence" against the documented 0.60 gate without such a guard/test.

### 2. Audio-only probe behavior — the regime this whole change targets — is still not validated with realistic probes
- **Severity:** Medium (benchmark/spec validity)
- **Where:** docs cite `..._final7.json` and `/tmp/auto_glossary_switch_fixture_probe_final7.json`; test log shows the `--with-probe` run uses synthetic `probe_for()` (0.9/0.8 matching the expected domain) and the real-source run uses `router_text_source=manifest_source` (text path, `with_probe=false`).
- **Why real:** The renormalization specifically changes the *audio-only, probe-driven* path. Neither cited "final7" run exercises it: one is text/keyword-driven, the other feeds hand-tuned single-domain synthetic probes. So "passed all four router-unit scenarios / zero false switches" is evidence for the text path, not for the newly-reachable renormalized probe path. The docs present these as the validation for this change.
- **How to verify:** Grep the eval — `probe_for()` returns a fixed 0.9/0.8 for the expected domain only; no realistic multi-domain probe or noisy ACL medicine-leakage case is fed.
- **Minimal fix direction:** Add a probe-only scenario with contested per-window scores (both domains scored, wrong domain occasionally higher) and assert no false switch; or explicitly state in the docs that audio-only renormalized-probe switching is unvalidated.

# Should fix

- **Doc path is non-host-qualified and inconsistent with its own command block (AGENTS.md violation).** `docs/adaptive_working_glossary.md` / `adaptive_working_glossary_eval.md` / `auto_glossary_routing_probe_20260707.md` prose cite `/mnt/data2/jiaxuanluo/rasst-demo/runtime/eval/..._final7.json`, but the command block immediately above writes to `/mnt/taurus/data2/jiaxuanluo/.../auto_glossary_switch_router_only_20260707.json`. AGENTS.md requires host-qualified Taurus paths and forbids bare `/mnt/...` non-Taurus paths in docs. The prose references a different, bare path than the documented command produces, so the "run" is not reproducible from the doc. Also `/tmp/...final7.json` is an ephemeral, non-reproducible location being cited as evidence.
- **Per-request window-matrix CPU transfer on the hot path even when unused.** `retrieval.py` `_retrieve_with_query_embeddings_sync` now does `valid_window_embs.detach().cpu().float()` and stores it on every `RetrievalResult` for every request every chunk, but it is only consumed when a gated probe fires. It is small (~N_windows×D), but it is an unconditional extra device→host copy added for all sessions/chunks against the concurrency positioning; gate it or accept it explicitly.
- **`_retrieve_slice_groups` gained a required positional param.** Both shown call sites (main + rescue) are updated; `omni.py` is truncated so a third caller cannot be excluded from the packet. A missing arg is a hard `TypeError` (not silent), so lower risk than the prior async concern, but confirm `grep -n "_retrieve_slice_groups" framework/agents/omni.py` shows only the two updated sites.
- **Reuse path silently ignores `_domain_probe_request`'s own audio window.** When the reused `query_embedding` (window matrix) is present, `_probe_window_embeddings_from_request` returns it and the probe never uses the request's `audio_buffer`/`current_*_sec`. Scoring the main-retrieval window is arguably correct, but the still-present `_domain_probe_request`/`_domain_probe_refresh_sec` audio machinery now only matters on the fallback (no-embedding) path — document this so the two windows are not assumed identical.

# Spec mismatch / surprise changes

- The score formula documented as fixed weights `0.60/0.25/0.10/0.05` now renormalizes over present signals, changing what `confidence`/`min_confidence` mean (item 1). Docs were updated to mention renormalization, but not the consequence that a single active signal saturates confidence and effectively bypasses the 0.60 gate.
- Commit title "Preserve window probe routing semantics" preserves the *MaxSim window* semantics of the probe (correctly), but simultaneously *changes* the router scoring/confidence semantics via renormalization; the title undersells the second, riskier change.

# Possible false positives (do not re-raise)

- The mean-pool-vs-MaxSim divergence flagged in the prior pass is genuinely fixed here: `query_window_embeddings` carries the already-`F.normalize`d per-window rows, and `_probe_window_embeddings_from_request` + `.max(dim=0)` yields true multi-window MaxSim consistent with the `_encode_probe_window_sync` fallback. `test_retrieval_plugin` now feeds a 2-window `[[1,0],[0,1]]` query and asserts the correct window-2 term, exercising this.
- `RetrievalResult.query_window_embeddings` inserted between existing fields is safe — all instantiations use keyword args.
- `WORKING_GLOSSARY_PRESETS` now including `common_10k` does not make it a routing target: `_domain_slices_for` still iterates `configured_working_presets(auto_glossary_presets)` (the 4 domain presets) and skips general/common; the addition only enables the mock base-slice path in `active_glossary._describe` and the base+overlay builder. `test_auto_active_retrieval_uses_common_base_plus_domain_overlay` confirms `[common_10k, nlp_core_10k]` / `[base, domain]`.
- Text-present path is ~unchanged: with all signals active the denominator ≈ 1.0, so renormalization is near-identity there.

# Final recommendation

Do not merge as-is. Before merge: (1) constrain the renormalized audio-only probe path so a lone normalized signal cannot clear `min_confidence` on the noisy probe evidence the repo's own doc distrusts — or add an audio-only, multi-domain, noisy-probe ACL-only regression asserting zero false switches; (2) stop citing text-path/synthetic-probe "final7" runs as validation for the renormalized audio-only probe behavior, and add a realistic probe-only case; (3) fix the non-host-qualified/self-inconsistent doc paths per AGENTS.md. The per-window MaxSim reuse (the headline fix) is sound; the renormalization is the item to gate. Items depending on the truncated `omni.py` (`_retrieve_slice_groups` caller completeness, `_probe_domain_scores` threading of the window embedding into the request) cannot be fully confirmed from the packet.
