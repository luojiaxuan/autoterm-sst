# Verdict
NEEDS_FIX

The change under review is `368262d → a5ed4ff` ("Add E2E generated-target glossary routing"): it fills in the previously no-op `_after_translation_tick` to write generated target text into `router_text_window`/`router_text_source`, drops the `common_terms` base slice (`base_slice: none`, `type: diagnostic`), simplifies `_autoterm_base_preset`, and adds Chinese topic keywords for nlp/medicine only. The base-slice removal and the external-source guard are correct and tested. But the generated-target routing path has a real functional gap for two of the four advertised domains, a precision/self-reinforcement risk that is untested, and its central correctness claim ("one routing tick behind") cannot be confirmed from the packet.

# Must fix before merge

### 1. Generated-target routing cannot switch to finance/legal for a Chinese target (silent dead path)
- **Severity:** High (spec mismatch / hidden functional gap)
- **Where:** `framework/agents/term_memory/domain_taxonomy.py` `DOMAIN_TOPIC_KEYWORDS`; consumed by `topic_keyword_scores` → `HybridWindowTopicRouter._score_hybrid`.
- **Why real:** The default `auto_glossary_presets` is `nlp_core_10k,medicine_core_10k,finance_core_10k,legal_core_10k`, and the new production signal is the *generated target* window, which for the default `English -> Chinese` pair is Chinese. This commit adds Chinese keywords to the `nlp` and `medicine` blocks only. The `finance` and `legal` blocks remain English-only (`\bmarket(s)?\b`, `\bcourt(s)?\b`, …). Python `\b`-anchored English regex never matches Chinese text, so `text_topic_score` for finance/legal is identically 0 on any Chinese generated target. The router therefore can never route *to* finance or legal via the generated-target path — the exact "deployable E2E route" this change is built to provide. The docs and paper wording claim the router "switches the active domain slice" among the domain inventory without stating that two of the four domains are unreachable through the shipping signal.
- **How to verify:** Feed a Chinese finance/legal generated-target window (e.g. 市场/股票/利率 or 法院/合同/管辖权) through `HybridWindowTopicRouter.observe(..., router_text_source="generated_target")` from a `finance`/`legal` starting state; confirm `text_topic_score` and any switch. There is no such test in `test_hybrid_window_topic_router.py` (only the nlp→medicine Chinese case).
- **Minimal fix direction:** Add Chinese finance/legal topic keywords symmetric with nlp/medicine, or explicitly scope the E2E generated-target claim/docs to the domains that actually have target-language coverage.

### 2. "One routing tick behind" guarantee is unverifiable and is the correctness hinge of the feature
- **Severity:** High if the call ordering is wrong (self-reinforcement becomes immediate)
- **Where:** `framework/agents/omni.py` `_after_translation_tick` (writes `router_text_window`) vs `_observe_active_glossary` (reads it and drives the router). The call site of `_after_translation_tick` within the streaming loop is **not in the packet** (`omni.py` truncated).
- **Why real:** The docs assert "The generated target window is one routing tick behind the current chunk, so a chunk's own translation cannot influence its own glossary retrieval." That property holds only if `_after_translation_tick` runs *after* `_observe_active_glossary` for the same chunk. If it runs before (or in the same pass that then re-routes), the current chunk's own translation — produced under the currently active glossary — feeds the domain decision for that chunk, closing a self-reinforcing loop with no guardrail. The method was previously a `return None` stub already invoked somewhere, so the call exists; the ordering does not.
- **How to verify:** `grep -n "_after_translation_tick" framework/agents/omni.py` and confirm it is invoked strictly after the per-chunk `_observe_active_glossary`, and that `session.history` at that point holds emitted *target translations* (not source/messages). Neither is confirmable from the packet.
- **Minimal fix direction:** None if ordering is already correct; if not, move the write after routing. Either way, add a test that asserts the current chunk's translation only affects the *next* chunk's `router_text_window`.

# Should fix

### 3. Generated target gets the lenient 2-window "with_text" threshold despite being a self-biased signal
- **Where:** `HybridWindowTopicRouter.observe` — `has_text = router_text and text_source != "none"`; generated_target ⇒ `has_text=True` ⇒ `min_consistent_windows_with_text` (2), and it bypasses the `_audio_probe_guard` (which only fires on `not has_text and has_probe`).
- **Why real:** Source transcript text is domain ground truth; generated target text is produced under the currently active (possibly wrong) glossary and can echo its domain vocabulary. Treating it with the same 2-window, no-raw-evidence-floor path as trusted source text is the weakest point for self-reinforcement the task explicitly flagged. With the renormalization introduced earlier on this branch, a single active text signal renormalizes `conf` toward its top-normalized value, so two windows of generic keyword hits can clear `min_confidence=0.60`.
- **Minimal fix direction:** Give `generated_target` its own (stricter) consistency/confidence treatment distinct from `manifest_source`, or require a raw-evidence floor before a generated-target-only signal can switch.

### 4. Several new Chinese keywords are high-frequency generic words → precision loss, biased toward the default NLP slice
- **Where:** `domain_taxonomy.py` new entries: nlp `注意力|嵌入|预训练|微调|提示` (0.9), `标注|解析|...` (1.0); medicine `试验|...` (1.0).
- **Why real:** `提示` (prompt/hint) and `解析`/`微调` are common in general and medical Chinese ("结果提示…"), and `试验` is generic. Because the default active slice is `nlp_core_10k`, spurious NLP keyword hits combine with the `min_current_margin` guard (which favors staying) to make it harder to *leave* NLP and easier to falsely score NLP on non-NLP audio. No test exercises noisy/realistic Chinese generated text — only one hand-built clean medicine sentence.

# Missing tests / benchmarks
- No agent-level test that a written `router_text_window`/`generated_target` actually flows through `_observe_active_glossary` into a router switch on the *following* chunk (the one-tick-behind contract, item 2).
- No finance/legal generated-target routing test (item 1) — the missing coverage hides that the path is dead there.
- No false-switch/self-reinforcement regression on realistic noisy Chinese generated text (item 3/4); `test_generated_target_chinese_topic_switches_after_two_windows` uses a keyword-dense clean sentence and cannot surface precision loss.
- No test that `session.history` semantics (target vs source) match what `_after_translation_tick` assumes.

# Performance risks
- None material. `_after_translation_tick` is a bounded string join over ≤`window` history items on the CPU path; not a retriever/GPU hot path.

# Spec mismatch / surprise changes
- Docs/README/paper wording present generated-target E2E routing as switching among the domain inventory, but the shipping Chinese-target signal only functions for nlp/medicine (item 1).
- `base_slice: none` + `common_terms type: diagnostic` correctly removes the common base from the active inventory (matches acceptance criteria), and `_active_retrieval_slices` returning only the domain slice is tested. Note `slice_role_for_preset("common_10k")` still returns `"base"` in the registry (`test_common_preset_maps_to_common_terms_slice`) — harmless only because `common_10k` is no longer in the active inventory; it is an internal inconsistency, not a bug in this path.
- Cited eval artifacts (`..._final8.json`) are now host-qualified per AGENTS.md, but are run results not present in the packet and remain unverifiable (pre-existing).

# Possible false positives
- The external-source guard (`current_source not in {"none","generated_target"}`) correctly preserves `manifest_source`/`streaming_asr` and is directly tested (`test_generated_target_translation_does_not_override_external_router_text`); the acceptance criterion "generated target must not override explicit external router_text" is met.
- The current-text dedup (`texts[-1] != current_text`) correctly avoids double-appending when `history` already contains the just-emitted translation; not a bug.
- Dropping the `type=="base"` scan in `_autoterm_base_preset` does not resurrect surprising legacy behavior: with `base_slice` present-and-empty or absent it returns the `"none"` default; `common_terms` is `diagnostic`, so it is never auto-selected as a base.
- `_meta_bool(enable_generated_target_text, True)` defaulting True is intentional and matches the yaml.

# Final recommendation
Do not merge as-is. Before merge: (1) add Chinese finance/legal topic keywords or scope the E2E claim to the covered domains; (2) confirm — with a test — that `_after_translation_tick` runs after per-chunk routing and that `session.history` holds target translations, so the one-tick-behind/no-self-influence property actually holds; (3) treat `generated_target` more conservatively than trusted source text (stricter consistency or a raw-evidence floor) and add a noisy-Chinese false-switch regression. Items 1 (partially), 2, and the `session.history` semantics cannot be fully confirmed from the packet because the `omni.py` call site is truncated.
